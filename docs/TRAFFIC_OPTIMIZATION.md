# Traffic Optimization

**Status:** Active  
**Scope:** Traffic-flow and authority reinforcement without route-family changes  
**Last updated:** 2026-03-16

## 1) Entry Page Discovery Audit

Top entry families reviewed:

- State authority and state-cost pages
- City electricity context pages
- Appliance cost pages
- Electricity bill estimator pages
- Comparison hubs

Findings:

- Core discovery pathways were already present, but estimator-to-appliance and city-to-appliance pathways were uneven across templates.
- Hub pages were strong but benefited from clearer links to benchmark bill and estimator clusters.
- Longtail pathway coverage existed in shared helpers, but needed an explicit appliance-estimator pathway section.
- Canonical ownership remained clean; no route-family or canonical changes were required.

## 2) Authority Hub Structure

Primary authority/discovery hubs:

- `/electricity-hubs`
- `/energy-comparison`
- `/electricity-usage`
- `/average-electricity-bill`
- `/electricity-bill-estimator`

Note on requested hub labels:

- Existing canonical architecture uses the above hubs and canonical clusters instead of standalone `/appliance-electricity-costs` or `/electricity-cost-by-state` route surfaces.
- Appliance and state-cost authority is reinforced through canonical clusters (`/cost-to-run/*`, `/electricity-cost/*`) and discovery hubs, not by introducing new route families.

Hub reinforcement strategy:

- Ensure each top-level hub links directly into canonical bill, estimator, calculator, usage, appliance, and comparison clusters.
- Keep links deterministic and rollout-safe.
- Preserve discovery intent for hubs and canonical ownership for destination families.

## 3) Longtail Traffic Pathway Design

Shared pathway expansion in `src/lib/longtail/internalLinks.ts` now reinforces:

- State -> appliance -> estimator pathways
- State -> appliance calculator pathways
- State -> appliance comparison slice pathways
- Existing state -> city and usage -> calculator pathways

Template-level reinforcement adds:

- City pages -> canonical appliance cost pages + representative estimator profile routes
- Estimator state/profile pages -> appliance cost and appliance calculator routes
- Energy comparison hub -> estimator/benchmark pathways for focus states

## 4) Discovery Graph Reinforcement

`public/discovery-graph.json` generation now includes stronger traffic edges:

- Hub -> canonical clusters:
  - `electricity-hubs -> appliance-cost`
  - `electricity-hubs -> electricity-usage-cost`
  - `electricity-hubs -> electricity-cost-comparison`
  - `electricity-hubs -> energy-comparison`
- Comparison pathways:
  - `energy-comparison -> electricity-bill-estimator`
  - `energy-comparison -> city-electricity-context`
- Estimator-appliance pathways:
  - `electricity-bill-estimator -> appliance-cost`
  - `appliance-cost -> electricity-bill-estimator`

All relationships remain deterministic and generated by `scripts/knowledge-build.ts`.

## 5) Verification Framework

Traffic optimization guardrails were added to existing verification scripts:

- `scripts/indexing-readiness.ts`
  - verifies `/electricity-hubs` and `/energy-comparison` expose expected canonical cluster links.
- `scripts/readiness-audit.ts`
  - adds traffic discovery checks for hub-to-cluster link presence.
- `scripts/verify-knowledge.js`
  - validates longtail pathway signals in shared linking helpers.
  - validates discovery-graph edges for estimator-to-appliance pathways.

## 6) Operating Rules

- Keep traffic optimization additive to existing canonical families.
- Keep rollout gating centralized in rollout helpers.
- Prefer shared link helper improvements over one-off route logic.
- Do not introduce new route families for authority labels that can be handled by existing canonical clusters and hubs.

## 7) Phase 3 (Implemented)

Traffic Optimization Phase 3 strengthened live-path discovery into the estimator profile pilot while keeping rollout boundaries explicit.

Implemented:

- Added allowlist-aware helper accessors in `billEstimator.ts` for state-scoped active profile discovery.
- Added a "Live profile pilot pathways" section on `/electricity-bill-estimator` that links only to explicitly active profile routes.
- Added an "Active profile pilot routes" cue on `/electricity-bill-estimator/{state}` when the state has allowlisted profile pages.
- Added state-pair page pathways from `/electricity-cost-comparison/{pair}` into the estimator family, with profile-route links shown only for states that are currently allowlisted.
- Added a direct estimator pathway in the intent-grouped section on `/{state}`.

Safety posture:

- No new route families.
- No inventory expansion (city, appliance × city, or estimator profile allowlist).
- No sitemap architecture changes.
- Canonical ownership unchanged.
- Deferred estimator profile inventory remains deferred beyond the active 8-key allowlist.
